# Retrieval Eval (RAG v1) — Design

**Дата:** 2026-06-19
**Status:** 📋 designed — pre-implementation
**Контур:** офлайн-eval для вибору embedding-моделі + репрезентації під майбутній RAG query-флоу
([`../architecture/2026-04-26-flow-production-rag.md`](../architecture/2026-04-26-flow-production-rag.md)).

---

## Мета

Відповісти на **одне** питання емпірично:

> Яку **embedding-модель** і **що саме ембедити** (`claim_text` / `situation` / `claim_text+situation`)
> брати для семантичного пошуку по прогнозах?

Рішення приймається за **відносним ранжуванням** комбінацій (хто кого переміг), не за
абсолютними цифрами. Абсолютна якість пошуку (launch-readiness) — поза скоупом, міряється
перед v1.5 (генерація відповідей).

## Чого v1 свідомо НЕ робить

- ❌ Жодного HTTP endpoint чи Telegram-бота (serving — пізніше).
- ❌ Жодної генерації відповідей LLM (це v1.5).
- ❌ Жодних фільтрів по `status`/`target_date`/`person` (це Phase 2 — hybrid search).
- ❌ У межах eval не чіпає прод-колонку `predictions.embedding` і не додає міграцій до alembic
  (зміна її розмірності під переможця — окремий follow-up по факту рішення, див. Скоуп).

## Рамка рішення (чому known-item достатньо)

Корпус однолюдний (Арестович) і тематично щільний → багато майже-дублікатів. Тому
**known-item** (чи джерельний прогноз потрапив у топ-k запиту, згенерованого з нього) міряє
*findability конкретного джерела*, а не повну релевантність. Це прийнятно, бо:

1. Для **відносного ранжування** комбінацій надлишковість корпусу — шум, спільний для всіх
   комбінацій → порядок «хто кращий» зберігається.
2. Надлишковість працює *на користь* рішенню про `situation`: два однакові за `claim_text`
   прогнози в різних обставинах розрізняє лише `situation` — там, де known-item «штрафує» за
   дублікати, `claim+situation` має підняти саме ціль.

Надлишковість **зменшуємо** (не усуваємо) хронологічним проріджуванням корпусу (див. нижче):
це знімає сплески-дублікати й робить метрику робочою. Залишкові рецидивні близнюки (та сама теза
через місяці) лишаються, але як **шум, спільний для всіх комбінацій** → відносний порядок не
ламають. Інші залишкові ризики гасяться: **leakage** — розділенням метрик по сім'ях запитів
(`source_field`); **зміщення синтетики** — ручним валідаційним зрізом.

**Нейтральність відбору:** корпус і цілі будуються **виключно з метаданих** (дата прогнозу +
`prediction_strength`/`prediction_value`) — жодного ембедера й жодної лексики. Тому набір не
фаворизує жодну модель-кандидата (зокрема baseline OpenAI 3-small): усі складають один і той
самий тест, якого ніхто з них не «креслив».

## Архітектура потоку

**Передумова (разово, поза скриптами eval):** прогнати **екстрактор + верифікатор** по історії
постів Арестовича → прогнози осідають у прод-Postgres із заповненими `prediction_strength` /
`prediction_value` (вони — вихід верифікатора, не екстрактора, тож без верифікації keep-правило
нижче не працює). Колонка `embedding` лишається null.

```
[прод-Postgres: predictions]  ← read-only, з strength/value
        │
        ▼
build_eval_corpus    → хронологічне проріджування (≥2 тиж між прогнозами,
        │              keep max strength/value у вікні) → eval-корпус (~150, поріг підкручуваний)
        ▼
build_query_gold.py   → семпл цілей (стратиф. по topic+часу) + 2 запити/ціль (LLM, temp 0)
        │                → retrieval_query_gold.json + ручний зріз retrieval_query_gold_manual.json
        ▼
embed_corpus.py       → для кожної (модель × репрезентація): embed корпус → retrieval_eval_emb
        │
        ▼
retrieval_eval.py     → для кожної комбінації × кожного gold-запиту: cosine top-k →
                        recall@k / MRR (розбивка по source_field) → markdown-звіт
```

Косинусний пошук — те саме `cosine_distance`-SQL, що в прод `PostgresVectorStore.search_similar`,
лише з фільтром `WHERE config = …` по eval-таблиці.

## Eval-таблиця (гнучка розмірність, поза alembic)

```sql
CREATE TABLE IF NOT EXISTS retrieval_eval_emb (
    prediction_id TEXT NOT NULL,
    config        TEXT NOT NULL,   -- "<model>__<repr>", напр. "e5-large__claim_situation"
    embedding     vector            -- БЕЗ фіксованої розмірності
);
```

- Безрозмірний `vector` дозволяє тримати різні моделі (1024/1536/3072) поруч; індекс не потрібен
  (точний косинус — це й є ground-truth ранжування).
- Створюється `embed_corpus.py` через `CREATE TABLE IF NOT EXISTS` — **не** в alembic, щоб не
  засмічувати прод-схему.
- `prediction_id` посилається на реальні прогнози прод-БД (FK логічно, не фізично — щоб eval не
  чіпав прод-констрейнти).

Прод-колонку `predictions.embedding (Vector(1536))` чіпаємо рівно раз — коли eval обере
переможця, одним міграційним рядком ставимо `Vector(dim_переможця)`.

## Gold-запити

### 1. Побудова eval-корпусу + відбір цілей (тільки метадані)
- **Eval-корпус (`build_eval_corpus`):** хронологічне проріджування прод-прогнозів — сортуємо за
  `prediction_date`, йдемо по таймлайну, лишаємо прогноз, лише якщо від попереднього залишеного
  минуло **≥2 тижні**; коли у вікно потрапляє кілька — лишаємо з найвищим `prediction_strength` /
  `prediction_value`. Поріг (2 тиж) підкручуваний: на історії Арестовича (~303 тиж) це дає ~150
  прогнозів; ослабити до 1 тиж → ~300.
- **Цілі:** семплимо ~80 із eval-корпусу, стратифіковано по `topic` **і** по періодах часу
  (рівномірне покриття тем і таймлайну).
- **Нейтральність:** і корпус, і цілі — суто з метаданих (дата + strength/value), без ембедера й
  лексики → жодна модель-кандидат не фаворизована.
- **Свідомі компроміси:** (а) рецидивні близнюки можуть лишитись — терпимо як спільний шум для
  відносного ранжування; (б) корпус скісний до high-strength і менший за прод — для *відносного*
  порівняння умови однакові для всіх моделей, але абсолютні цифри з нього не беремо (це й не
  мета v1).

### 2. Генерація запитів (LLM, Flash Lite, temp 0 — детерміновано)
На кожну ціль — **два** записи:

```json
{ "query": "...", "target_id": "<Prediction.id>", "source_field": "claim_text" }
{ "query": "...", "target_id": "<Prediction.id>", "source_field": "situation" }
```

- `source_field = claim_text` → content-запит (про ЩО спрогнозовано), генерується з `claim_text`.
- `source_field = situation` → context-запит (про ОБСТАВИНИ/період), генерується з `situation`.
- **Анти-leakage в промпті:** писати як пересічний користувач, природною UK/RU, **не копіювати**
  характерні слова/числа/назви з джерела, узагальнювати.
- Eval-специфічний промпт допустимий: у генерації запитів немає прод-аналога, тож конвенція
  «не форкай прод-промпт» не порушується.

### 3. Ручний валідаційний зріз
- ~20 запитів, написаних людиною, кожен з ручною розміткою релевантних `target_id` (можливо
  кілька). Файл `retrieval_query_gold_manual.json`.
- Використовується лише для перевірки, що синтетичний порядок тримається на живому формулюванні —
  **не** входить в основне ранжування.

## Комбінації (sweep)

- **Репрезентації (3):** `claim_text` · `situation` · `claim_text + "\n" + situation`.
- **Моделі (≈3, фіналізуються окремим screening по MMTEB UK/RU):** baseline OpenAI
  `text-embedding-3-small` + 2 мультимовні кандидати, доступні через LiteLLM.
- Усі — через прод `EmbeddingClient`, лише підміна параметра моделі.

## Метрики (відносні, розбивка по `source_field`)

- **recall@k**, k ∈ {1, 5, 10, 20} — частка запитів, де `target_id` у топ-k. **Головна.**
- **MRR** — діагностична (середнє 1/позиція); рахуємо й показуємо, але рішення не визначає.
- Кожна — окремо для `claim_text`-запитів, `situation`-запитів і сумарно.

## Правило вибору переможця

- **Модель:** найвищий сумарний `recall@10`. Тайбрейкер — `recall@1`.
- **Репрезентація:** беремо `claim+situation`, **якщо** вона не програє кращій одиночній
  репрезентації на *жодній* із двох сімей більше ніж на поріг (напр. 3 в.п.). Тобто concat має
  бути «не гірше за обидві», а не лише «краще в середньому».
- **Ручний зріз:** якщо суперечить синтетичному порядку — синтетиці не віримо, розбираємось.

## Компоненти та файли

```
scripts/retrieval/
  __init__.py
  eval_store.py         # EVAL_TABLE_DDL + EvalEmbStore Protocol + PostgresEvalEmbStore (write/search)
  build_eval_corpus.py  # хронологічне проріджування ≥2тиж + keep max strength/value
  build_query_gold.py   # семпл цілей (topic+час) + генерація 2 запитів + анти-leakage промпт
  embed_corpus.py       # sweep: ensure_table + наповнення retrieval_eval_emb
  retrieval_eval.py     # cosine top-k + recall@k/MRR + markdown-звіт
scripts/data/
  retrieval_query_gold.json         # синтетичні запити
  retrieval_query_gold_manual.json  # ручний зріз (пише людина)
scripts/outputs/retrieval_eval/
  retrieval_eval_report.md
  per_config/<model>__<repr>.json
```

Перевикористання прод-коду: `EmbeddingClient` (`llm/embedding.py`), `cosine_distance`-патерн
(`storage/postgres.py`), репозиторій для читання корпусу прогнозів.

## Тестування

- Unit (in-memory, без БД/мережі, за конвенцією проєкту):
  - хронологічне проріджування (≥2тиж між залишеними; у вікні лишається max strength/value) —
    на синтетичному наборі дат;
  - стратифікований семпл цілей (покриває topic + періоди);
  - парсинг/валідність gold (рівно 2 записи на ціль; `source_field` ∈ {claim_text, situation});
  - підрахунок recall@k / MRR на відомих ранжуваннях (фікстури);
  - формування тексту репрезентації (3 варіанти).
- Інтеграційно (real Postgres + LLM, опціонально, як `integration_smoke`): малий прогін
  `embed_corpus → retrieval_eval` на ~10 прогнозах, перевірка, що звіт генерується.

## Скоуп

**In:** офлайн eval-скрипти, хронологічна побудова eval-корпусу, eval-таблиця, gold-генерація,
sweep по 3 репрезентаціях × ~3 моделях, recall@k/MRR-звіт, ручний зріз, вибір переможця.

**Передумова (поза eval-скриптами):** прогін екстрактора + верифікатора по історії постів, щоб у
прод-БД були прогнози із заповненими strength/value.

**Out (deferred):** HTTP/bot serving, генерація відповідей (v1.5), метадані-фільтри/hybrid (Phase 2),
абсолютна релевантність / pooled LLM-judge (перед v1.5), фінальний screening моделей по MMTEB
(окремий короткий крок перед запуском sweep), зміна прод-колонки під переможця (1 міграція по
факту рішення).

## Очікуваний результат

Markdown-звіт з таблицею комбінація × {recall@1/5/10/20, MRR} × {claim_text / situation / overall}
і явним переможцем (модель + репрезентація), підтвердженим ручним зрізом. Це фіксує два параметри
для майбутнього RAG-флоу й розблоковує v1.5 (генерацію).
